1. **Description field**: Change from summary ("Validate ComfyUI workflow JSON...") to trigger ("When the user has a workflow JSON and wants to check compliance, submission readiness, or cloud compatibility"). The description answers "when should I fire?" not "what do I do."

**Gotchas files**: Create gotchas.md in each skill folder documenting known failure points discovered during Phases 1-5. Extract from STATE.md decisions and codebase patterns:

**comfy-discover/gotchas.md:**
- GitHub API rate limits: 60 req/hr unauthenticated. Set GITHUB_TOKEN env var for higher limits.
- httpx needs follow_redirects=True for GitHub API (301 on /contents endpoint)
- Registry "tags" field is empty for most packs — category filtering uses keyword matching instead
- Cache TTLs: 1hr browsing, 15min search, 24hr specs. Use --refresh to bypass.
- search_by_type uses two-step: search packs then verify I/O via comfy-nodes API

**comfy-templates/gotchas.md:**
- Fuzzy matching on index keys when exact match returns nothing — may return unexpected results
- fetch_all_nodes uses `pages` param — don't assume single-page responses
- Gap scoring formula: log10(downloads) * (1 + log2(stars) * 0.5) — balances volume with community signal
- Coverage % is pack-level, not node-level

**comfy-validate/gotchas.md:**
- UUID-style node types (>30 chars with hyphens) are skipped in custom node detection — these are subgraph references, not custom nodes
- API node detection runs separate from RULE_REGISTRY to keep auth concerns distinct
- Note color darkness check uses #0 prefix heuristic — may miss edge cases
- Lenient mode skips info-level rules that always fire on minimal workflows

**comfy-compose/gotchas.md:**
- NodeSpecCache is pass-through: Claude must fetch MCP specs and pass raw dicts in — no auto-fetching
- from_json supports both array and object link formats (ComfyUI versions differ)
- swap_node only removes connections when spec is provided (graceful degradation without spec)
- auto_layout uses DFS longest-path layer assignment — may produce suboptimal layouts for highly connected graphs
- save_workflow uses lenient validation by default since composed workflows are drafts

**comfy-document/gotchas.md:**
- Model detection uses Load substring in node type plus file extension heuristic — may miss non-standard loaders
- extract_node_types is reused from src.templates.fetch for consistency
- Format gating: rejects API-format JSON (must have nodes array, not string keys)
- Subgraph IO extraction looks inside definitions — but nested subgraphs are not recursed

**comfy-flow/gotchas.md:**
- Phase transitions are pure functions — advance_phase returns next phase, doesn't mutate session
- Validation gate blocks advancement to document phase — user can override with "skip validation"
- Session state is in-memory only — not persisted across Claude Code conversations
- CLI prints guidance message (not error) when no --scaffold/--file given

# ComfyUI Template Agent

Claude Code agent toolkit for ComfyUI template creation. Six skills guide template creators from node discovery through submission documentation.

## Skills

| Skill | Purpose |
|-------|---------|
| comfy-discover | Node discovery from ComfyUI registry |
| comfy-templates | Template library browsing and gap analysis |
| comfy-validate | Workflow guideline validation |
| comfy-compose | Workflow composition and scaffolding |
| comfy-document | Submission documentation generation |
| comfy-flow | Guided end-to-end template creation |

## Development

```bash
pytest                                    # Run tests
python -m src.registry.highlights trending  # Test discovery
python -m src.validator.validate --file workflow.json  # Test validation
```

## Architecture

- `src/shared/` — HTTP client, caching, format detection
- `src/registry/` — Node discovery (highlights, search, spec)
- `src/templates/` — Template library (fetch, search, cross_ref, coverage)
- `src/validator/` — Validation engine (rules, engine, validate CLI)
- `src/composer/` — Workflow composition (graph, scaffold, layout, compose CLI)
- `src/document/` — Documentation generation (metadata, notion, generate CLI, orchestrator)
- `data/` — Static data (core_nodes.json, guidelines.json, api_nodes.json)

## Conventions

- Pydantic models for all data structures
- Module-level DiskCache singletons for HTTP caching
- httpx with follow_redirects=True for all HTTP requests
- Tests in tests/ mirroring src/ structure
- CLI entry points via `python -m src.module.script`
